ALUA EFI LOADER INTERFACE - VERSION 0.4.5

Alua for EFI EDK2-Native

High-Performance, Register-based Scripting Engine optimized for UEFI MS ABI.

核心技術 (Firmware Core)

  • MS ABI Support: 完全符合 UEFI 調用慣例，與 edk2 深度整合。
  • LibC Integration: 已手動補足韌體環境下缺失的標準 C 庫支持。
  • Register-based: 針對寄存器架構設計，非傳統棧式機，提升指令執行效率。

> 提示：由於 RootFS 深度依賴 UEFI API (efiP)，此版本被列為核心基礎版，永不廢除。

指令集架構 (ISA - EFI Optimization)

32-bit 定長指令，確保在 Pre-Boot 內存受限環境下的解碼效率：

分類 指令 (OpCodes) EFI 運作說明
Data LOADK, MOVE 常量與寄存器操作
Math ADD, SUB, MUL, DIV 純整數運算 (無浮點數依賴)
Flow JMP, EQ, TEST 條件跳轉與判斷
System PRINT, CALL 調用 UEFI ConsoleOut 或 Protocol

部署方法 (EDK2 / QEMU)

請將編譯後的 BOOTX64.EFI 放入 ESP 分區。建議測試環境：

# 1. 準備文件結構
mkdir -p qemu_root/EFI/BOOT
cp -r ./ROOTFS/* ./qemu_root/
cp EFI/BOOT/BOOTX64.EFI ./qemu_root/EFI/BOOT/

# 2. 啟動 QEMU (需指定 OVMF)
qemu-system-x86_64 -machine q35 -pflash /path/to/ovmf.fd \
  -drive format=raw,file=fat:rw:qemu_root -m 256

EFI 常見問題

為什麼 EFI 版不會被廢除？

因為 Alua EFI 作為 AOS 的第一階段引導核心，承載了大量與硬體直接對接的 API。若廢除此版，將導致整個 AOS 生態系無法在原生韌體上啟動。

APP 相容性注意

若 App 使用了 efiP (EFI Protocol) 專屬模塊，則無法在其他非 EFI 環境（如 Linux）運行。請開發者務必確認 API 調用範圍。
